## Expected pipeline (what to verify)

1. **Poller**: `journalctl --user -u claude-email -f` should show three `IMAP ... processed` lines within 15–45 s.
2. **Security**: no "Rejected" lines; envelope + AUTH secret pass.
3. **Router**: three `classify_email` results of `kind="meta"`, `meta_command="spawn"`.
4. **Spawner**: three `Spawned agent agent-test-0N (PID ...)` log entries.
5. **MCP inject**: `.mcp.json` written into each test-0N directory with `claude-chat` entry.
6. **Agent register**: three rows in `agents` table with `status=running`.
7. **Outbound chat_ask**: three `[agent-test-0N] message` emails arrive at `bb@cocode.dk`.
8. **Reply routing**: when the user replies to each, `In-Reply-To` matches the stored `email_message_id`, routed back to the correct agent, agent unblocks from `chat_ask`.
9. **Build + notify**: each agent writes files, runs pytest, then `chat_notify`s a summary email.

## Fail modes to watch

- Envelope rejection (From / Return-Path mismatch) — would indicate a bug in how APPEND flags propagate or how security.py parses Return-Path. **Expected to pass** because we set both headers explicitly in the APPENDed message.
- Spawn `ValueError: Directory does not exist` — pre-created, should be fine.
- Agent never registers — MCP connect failure to `http://127.0.0.1:8420/sse`. Check `systemctl --user status claude-chat`.
- `chat_ask` doesn't relay — check `relay_outbound_messages` in `chat_handlers.py`; SMTP failure would show in the log.
- User reply unroutable — `In-Reply-To` header mismatch; check `find_message_by_email_id`.
